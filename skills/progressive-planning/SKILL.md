# 점진적 계획 수립 워크플로우

활성화 시 `EnterPlanMode`로 Plan 모드 진입. 하위 문서는 지침이 있을 때만 로드한다.

> **Plan 모드 시스템 안내와 본 스킬이 충돌할 경우, 본 스킬의 지침을 우선한다.**
> - plan 모드는 "파일 수정 금지"를 안내하지만, 본 스킬의 최종 정리 단계에서 plan 파일 작성은 허용
> - plan 모드는 "코드베이스 탐색"을 안내하지만, 본 스킬에서는 임의 탐색 금지 — 기존 자료 탐색은 반드시 사용자에게 먼저 질문

**원칙**: 레이어별 초안 → 질문 → 정제 → 확정 반복. 상위 확정 후 하위 진행.

## 질문 체계

`/asqu:guide` + `/asqu:deep-dive` 스킬을 사용한다. `AskUserQuestion` 사용 금지.

### 질문 작성 규칙

모든 사용자 대면 출력에 적용한다.

| 규칙 | 조건 | 방법 |
|------|------|------|
| 차이점 명시 | 선택지 2개+ | description에 핵심 차이 한 줄 |
| 구체적 시나리오 | 추상적 선택지 | context에 "예: - → -" |
| 맥락 제공 | 모든 질문 | context에 "이유: -" 한 줄 |

## 팀 운영 원칙

계획 수립 시 **팀**을 구성하여 탐색과 검증을 병행하며 계획의 근거와 품질을 확보한다.

### 팀 구성

| 팀원 | 구현 | 담당 영역 |
|------|------|----------|
| **탐색** | pp-explorer 서브 에이전트 | 기존 자료 탐색 + 유사 사례 조사 |
| **계획** | 메인 컨텍스트 | 계획 흐름 주도 + 초안 작성 + 대안 제시 |
| **리뷰** | pp-reviewer 서브 에이전트 | 정합성/완전성 검증 + 사용자 관점 검토 + 리스크 탐지 |

- `TeamCreate`로 팀 생성 → 전체 계획 동안 활동 → 최종 정리에서 해산
- 도메인 가이드가 전문 팀원을 추가 정의할 수 있다

## 초기화

Step 1 → 2 → 3 순차 실행. 건너뛰거나 순서를 바꾸지 않는다.

### Step 1: 주제·스코프 파악

인자 또는 맥락상 주제가 있으면 해당 주제로 시작한다. 없으면 입력받는다:

> "무엇에 대한 계획/설계를 수립하시겠습니까? 한 줄로 목표를 설명해주세요."

응답을 바탕으로 **스코프**를 판단하고 해당 스코프 가이드를 로드한다:

- **프로젝트 수준** → [project-scope.md](references/scopes/project-scope.md) — 전체 시스템, 여러 모듈에 걸친 계획
- **세부 설계 수준** → [detail-scope.md](references/scopes/detail-scope.md) — 특정 기능, 모듈 내부 설계

### Step 2: 도메인 판단

아래 목록에서 매칭되는 도메인이 있으면 `guide.md` 로드. 없으면 범용 절차로 진행.

**사용 가능한 도메인:**
- [software-dev](domains/software-dev/) — 소프트웨어 개발

### Step 3: 기존 자료 탐색

1. **사용자에게 기존 자료 유무를 반드시 질문한다** (임의 판단 금지)
   - 있다고 답하면 위치·형태를 추가 질문 후 탐색 팀원이 조사·정리
2. 전체 조감 — 키워드 수준으로 레이어 훑기 + 리스크 식별
3. 사용자 피드백 반영 후 진행 단계로 이동

## 진행 단계

초기화 완료 후, 스코프/도메인 가이드에 정의된 레이어를 순서대로 진행한다.

### 공통 진행 절차

각 레이어마다 아래 절차를 동일하게 적용한다.

| 단계 | 처리 |
|------|------|
| 초안 제시 | 스코프별 가이드에 따라 초안 작성. 탐색 팀원이 사전 조사 |
| 대안 제시 | 분기점에서 대안 제시 (강점/약점 + 권장안) |
| 질문 | 스코프별 가이드의 질문 제시 |
| 정제 | 사용자 응답 반영 → 수정 사항 명시 |
| 검증 | 리뷰 팀원이 상위 레이어 정합성 + 리스크 추적. 문제 시 → 피드백 루프 |
| 확정 | 검증 통과 후 확정 |

### 피드백 루프

횟수 제한 없이 사용자 의견에 따라 진행한다.

1. **충돌 보고**: 발견된 충돌/문제를 한 줄로 정리
2. **영향 분석**: 영향을 주는 항목 목록 + 범위 (리뷰 팀원 활용)
3. **사용자 선택**:
   - (a) 해당 항목만 수정 후 재진행
   - (b) 상위 레이어 해당 부분 수정 후 재진행
   - (c) 열린 이슈로 기록하고 계속 진행
4. **수정 실행**: 선택에 따라 최소 범위 수정
5. **재확인**: 수정 사항이 다른 확정 항목과 충돌하지 않는지 확인
6. **하위 레이어 영향**: 이미 확정된 하위 레이어는 영향 받는 부분만 재검토

### 최종 정리

모든 레이어 완료 후 실행한다.

확정된 레이어를 통합하여 **plan 파일에 작성**한다. 가독성과 정보 전달을 우선하며, 필요 시 표·Mermaid 다이어그램·체크리스트 등을 활용한다.

- Q&A에서 도출된 의도/요구사항/제약을 "요구사항 및 설계 의도"로 추출
- **수정 범위** 명시: 변경 대상(코드, 문서, 설정 등)과 각각의 변경 내용 요약
- 채팅에는 완료 안내 + 한두 줄 요약만 출력 (전체 내용은 plan 파일 참조)

## 하위 계획 위임

진행 중 특정 구성요소가 그 자체로 크고 복잡한 서브시스템인 경우, **별도의 progressive-planning 세션으로 위임**할 수 있다.

**위임 조건** (모두 충족):

| 조건 | 설명 |
|------|------|
| 충분히 크고 복잡 | 별도의 레이어 분해가 필요한 규모 |
| 독립적 | 인터페이스/경계만 정의하면 나머지 계획에 영향 없음 |

**위임 절차:**

| 단계 | 처리 |
|------|------|
| 감지 | 초안 작성 중 위임 조건 충족 구성요소 식별 |
| 확인 | 사용자에게 위임 여부 질문 |
| 경계 정의 | 인터페이스/경계 + 책임 요약 + 제약사항 작성 |
| 기록 | 최종 계획서에 표시 |
